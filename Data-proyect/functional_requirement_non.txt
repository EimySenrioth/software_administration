RNF-011: The system must self-destruct if it detects that it is being analyzed in a virtual or debugging environment

1. GENERAL INFORMATION
Field                  Detail
Requirement ID         REQ-SEC-013
Name                   Self-Destruction upon Detection of Virtual or Debugging Environment (Anti-Forensics)
Module                 Endpoint Security / Application Protection (PC and Mobile)
Priority               Critical
Status                 Sensitive / In definition
Version                1.0
Date                   24/05/2024

2. DESCRIPTION
The system must implement an active defense mechanism (Kill Switch) that continuously monitors the application's execution environment. If indicators are detected that the software is running inside a virtual machine (VM), emulator, sandbox environment, or if a debugger is attached to the process, the system must activate a self-destruct protocol.

3. PROCESS

1. Environment Detection (Environment Check):

   * On PC: Check for typical virtualization artifacts (specific VM video drivers, virtual network card MAC addresses, presence of hypervisor services, constant CPU temperature, forcibly synchronized system clock).

   * On Mobile: Detect if the device is rooted/jailbroken.

4. OUTPUTS

* Session Termination

5. SOURCE
List of running processes

6. RESTRICTIONS

* Local Data Loss: Self-destruction implies the complete loss of all information. The user must be aware that running in unsafe environments results in the annihilation of local data.

RNF-013: Protection against memory dumping

1. GENERAL INFORMATION
Field                  Detail
Requirement ID         REQ-SEC-015
Name                   Protection against Memory Dumping (Anti-Memory Dumping)
Module                 Application Security (Runtime Protection)
Priority               High
Status                 Defined
Version                1.0
Date                   24/05/2024

2. DESCRIPTION
The objective is to prevent attackers from performing a "memory dump" to recover information.

3. PROCESS
Anti-Dump (Fooling the Dumper):
Modify the executable headers in memory (PE Header / Mach-O) to break the structure expected by dumping tools, causing the generated dump file to be corrupted or unable to be opened by standard analyzers.

4. OUTPUTS
Protected memory

5. SOURCE
Memory

6. RESTRICTIONS
Performance Overhead: Constantly encrypting and decrypting data in memory consumes additional CPU cycles. This protection should be applied selectively only to the most critical data (secrets).


RNF-014: Daily renewal of encryption keys

1. GENERAL INFORMATION
Field                  Detail
Requirement ID         REQ-SEC-016
Name                   Daily Rotation of Encryption Keys (Key Rotation)
Module                 Cryptographic Key Management (KMS)
Priority               High
Status                 In definition
Version                1.0
Date                   24/05/2024

2. DESCRIPTION
The system must implement an automatic cryptographic key rotation policy.

3. PROCESS

Automatic Key Generation (Key Scheduling): A secure service (or the KMS - Key Management Service module) must automatically generate a new key pair or symmetric key daily without manual intervention. The old key set must be immediately archived.

Progressive Re-encryption (Data Re-encryption):
For new data: Use the keys of the "current day" (Day N) for encryption.
For existing data (encrypted with Day N-1 keys): The system must initiate a background process to decrypt records with the old key and re-encrypt them with the new key, or maintain an envelope encryption system where the daily "master key" protects the data keys.

4. OUTPUTS

* New Daily Key

5. SOURCE
System clock (UTC)

6. RESTRICTIONS
Only for important data.
